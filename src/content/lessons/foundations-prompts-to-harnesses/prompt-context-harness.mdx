Most people start with **prompt engineering** — tweaking the wording of one instruction. That's
the smallest lever in the system. The leverage lives one and two layers out.

<ConceptCard title="Three nested scopes" tag="topic 1">

- **Prompt** — the wording of a single instruction.
- **Context** — the *entire token set* in the window on a given step: system prompt, tool
  definitions, retrieved data, conversation history, memory.
- **Harness** — the code and control flow *around* the model: the loop, tool execution, output
  parsing, verification, state, recovery, and budgets.

Karpathy's framing — **LLM-as-OS**: the model is the **CPU**, the context window is **RAM**, and
the harness is the **OS** that decides what to load and when to call tools.

</ConceptCard>

<KeyInsightCallout>
The prompt is the smallest lever; the model is a component, not the product — you ship a
**system**, not a string.
</KeyInsightCallout>

## Harness engineering

The harness is the program that turns a stateless next-token predictor into a *task-completer*.
The same model scores very differently depending on the loop you wrap around it — add test
execution and failure feedback, and capability jumps without touching the weights.

<KeyInsightCallout>
Hold the model fixed and a better loop — tools, feedback, verification, budget control — can
outperform any amount of prompt rewriting.
</KeyInsightCallout>

> **Accuracy note.** Context-window sizes vary (~200K is common; some models offer ~1M modes) —
