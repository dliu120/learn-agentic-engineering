<ConceptCard title="A trace is a tree of spans" tag="topic 18">

In OpenTelemetry-style terms, a **trace** is one request end-to-end; a **span** is one timed unit — an LLM call, a retrieval, a tool invocation — nesting into a tree. GenAI semantic conventions standardize attributes like model, token counts, and latency. (These conventions were still **experimental in early 2026** — verify attribute names against the current spec.)

</ConceptCard>

<KeyInsightCallout>
The unit of debugging is the trace, not the log line — without span structure you can't tell whether a bad answer came from retrieval, prompt, or model.
</KeyInsightCallout>

## LLM-specific telemetry

Classic monitoring watches HTTP status, CPU, memory. LLM apps fail — and bleed money — elsewhere:

<TradeoffTable headers={["Signal","Why it's first-class"]} rows={[["Output tokens","Pricier than input and dominate latency (autoregressive)"],["TTFT / ITL / tail","p99 can be terrible behind a healthy p50"],["Semantic errors","Invalid JSON, schema violations, refusals, truncation, tool-call failures — all inside a 200"]]} />

<KeyInsightCallout>
LLM observability must treat "valid JSON?", output-token count, and refusals as first-class signals — that's where LLM apps fail and bleed money.
</KeyInsightCallout>

## Drift and closing the loop

**Data drift** (input distribution shifts), **concept drift** (the right answer changes), and **model drift** (a silent provider upgrade) all degrade quality quietly. Detect via input/output/score distribution monitoring — PSI, KL, KS — plus online sampled judge evals. PSI thresholds (~0.1 / ~0.25) are a borrowed heuristic, not a law.

<KeyInsightCallout>
Pin model snapshots and alert on drift — the scariest regression is the one you never deployed.
</KeyInsightCallout>
